## 3. 任务卡详情

### 3.1 S2-001 接入真实 Markdown 解析器
- 任务ID：S2-001
- 任务名称：接入真实 Markdown 解析器
- 主模块：Muse.Editor.Rendering
- 关联阶段/Sprint：Sprint 2 Day 1
- 前置依赖：S1-003

#### 目标与边界
- 需求目标：将当前占位解析逻辑替换为真实 Markdown 解析能力，保持 Rendering DTO 输出契约不变。
- 不做范围：不改 UI 布局，不改 Core 契约，不引入导出逻辑。
- 输入输出契约：
  - 输入：RenderRequest（Markdown 文本、模式、主题）。
  - 输出：RenderingResult（RenderedBlock 列表 + Diagnostic）。
- 影响范围：Muse.Editor.Rendering 及对应测试工程。

#### 实现约束
- 架构约束：UI 层不得直接调用解析器库，必须通过 MarkdownEngineAdapter。
- 禁止事项：禁止在此任务里改动 MainViewModel 路由状态机。
- 依赖新增：如引入第三方解析器，必须在变更说明中明确版本与选择理由。
- 是否允许改动公共契约：允许微调 Rendering 内部实现，不允许破坏 DTO 兼容。

#### 风险与回滚
- 风险点：真实解析器输出与现有分类模型不一致，导致测试大面积失败。
- 回滚条件：解析接入后基础用例通过率低于 80%。
- 回滚方案：保留适配层接口，切回占位解析实现并逐步迁移测试。

#### 验证与交付
- 测试用例：
  - 标题、列表、代码块、表格、引用、任务列表分类。
  - 空文档、超长文档、异常输入诊断路径。
- 验收标准：
  - Rendering 测试全通过。
  - 新增解析能力后，旧契约测试不回归。
- 需要更新的文档：
  - files/实施计划.md
  - files/项目架构文档.md
  - files/变更说明模板.md（按实例补一份实际说明）
- 完成后执行的门禁命令：
  - scripts/Invoke-QualityGate.ps1

### 3.2 S2-002 建立渲染映射层
- 任务ID：S2-002
- 任务名称：建立渲染映射层
- 主模块：Muse（UI 宿主层）
- 关联阶段/Sprint：Sprint 2 Day 1
- 前置依赖：S2-001、S1-004

#### 目标与边界
- 需求目标：建立 UI 对 RenderingResult 的映射入口，让编辑模式与分屏预览消费同一数据源。
- 不做范围：不做视觉打磨，不做高级交互（目录联动、代码块工具栏）。
- 输入输出契约：
  - 输入：RenderingResult。
  - 输出：可用于 UI 展示的轻量 ViewState。
- 影响范围：MainViewModel、MainView 绑定结构、UI 映射服务。

#### 实现约束
- 架构约束：UI 只依赖 Rendering DTO，不依赖解析器实现。
- 禁止事项：禁止把 Markdown 解析逻辑塞进 ViewModel。
- 依赖新增：无新增第三方依赖。
- 是否允许改动公共契约：不改 Core/Rendering 公共契约，仅新增映射层。

#### 风险与回滚
- 风险点：映射层设计不当导致后续增量渲染难以接入。
- 回滚条件：UI 与 Rendering 耦合回归（出现直接调用解析器实现）。
- 回滚方案：回退到占位预览文本绑定，保留映射层接口。

#### 验证与交付
- 测试用例：
  - 编辑模式输入后预览区同步更新。
  - 分屏左右/上下切换后预览仍可用。
  - 阅读模式展示来自同一映射源。
- 验收标准：
  - 三模式预览来源统一。
  - UI 不直接依赖解析器具体类型。
- 需要更新的文档：
  - files/实施计划.md
  - files/项目架构文档.md
- 完成后执行的门禁命令：
  - scripts/Invoke-QualityGate.ps1

### 3.3 S2-003 工作区最小能力骨架
- 任务ID：S2-003
- 任务名称：工作区最小能力骨架
- 主模块：Muse.Workspace
- 关联阶段/Sprint：Sprint 2 Day 1
- 前置依赖：S1-001

#### 目标与边界
- 需求目标：落地文件树、多标签、自动保存的最小状态模型与接口，供 UI 后续接入。
- 不做范围：不做完整文件系统 UI，不做复杂冲突处理。
- 输入输出契约：
  - 输入：工作目录路径、文档打开/切换事件、草稿变更事件。
  - 输出：文件树状态、多标签状态、自动保存调度状态。
- 影响范围：Muse.Workspace 及对应测试（若新增）。

#### 实现约束
- 架构约束：Workspace 只负责状态与服务，不依赖 UI 控件类型。
- 禁止事项：禁止直接写死绝对路径或平台特定路径分隔行为。
- 依赖新增：如引入文件监听库，必须先在任务卡评审中确认。
- 是否允许改动公共契约：允许新增 Workspace 内部契约，不破坏现有模块边界。

#### 风险与回滚
- 风险点：状态模型设计过重，影响后续接入成本。
- 回滚条件：模型难以序列化或测试复杂度失控。
- 回滚方案：回退到最小字段集，分离保存调度逻辑。

#### 验证与交付
- 测试用例：
  - 打开多个文档标签并切换状态。
  - 标记脏状态并触发自动保存调度。
  - 工作目录切换后文件树状态重建。
- 验收标准：
  - 状态模型可测试、可序列化。
  - 后续 UI 可通过契约接入而不改底层模型。
- 需要更新的文档：
  - files/实施计划.md
  - files/项目架构文档.md
- 完成后执行的门禁命令：
  - scripts/Invoke-QualityGate.ps1
