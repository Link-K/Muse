# S2-006 — 标签拖拽与重排（实现说明）

日期: 2026-06-02

概述
- 本次任务实现了工作区标签（tab）的拖拽重排、后端持久化（`tabs.json`）与基于持久化顺序的恢复逻辑，并修复了模板命令绑定引起的 XAML 可靠性问题。

实现要点
- 前端 (UI)
  - 在 `MainView.axaml` 中将标签模板的命令绑定改为直接绑定到每个 tab 的命令代理（`ActivateCommand` / `CloseCommand`），避免 DataTemplate 中对根 DataContext 的不稳定引用。
  - 使用 pointer-based 事件（已有）触发 `MainViewModel.ReorderTabs(documentId, newIndex)` 来发起重排。

- ViewModel
  - 在 `WorkspaceTabViewModel` 中新增 `ICommand ActivateCommand` 与 `ICommand CloseCommand`，增加构造重载以注入命令代理。
  - 在 `MainViewModel.SyncWorkspaceState()` 中为每个打开标签生成 `ActionCommand`（代理调用 `ActivateTab` / `CloseTab`），并通过新的构造函数注入到 `WorkspaceTabViewModel`，从而允许模板内直接绑定。

- 后端 (Workspace 服务)
  - `InMemoryWorkspaceService.MoveTab(documentId, newIndex)`：实现内存重排并在成功后将当前顺序保存到 `.muse/settings/tabs.json`。
  - `OpenWorkspace(root)`：增强恢复逻辑：
    - 优先从 `.muse/recovery/*.json` snapshots 加载（已有）。
    - 当无 recovery snapshots 但存在 persisted `tabs.json` 时，会基于 persisted ids 构造 `OpenTabs`（仅包含仍然存在的文件），从而支持 persisted-only 恢复场景。

注意事项与边缘情况
- persisted ids 指向已删除或不可访问的文件时，恢复逻辑会忽略该 id（不抛异常）。
- recovery snapshot 与 persisted order 同时存在时，以 snapshot 为基础并按 persisted 顺序重排 snapshot 中存在的项；未出现在 persisted order 的 snapshot 会被追加在末尾。
- XAML 绑定方案已改为每-tab 命令代理，避免在 DataTemplate 中对 `ElementName` / `x:Reference` 的依赖，从而提高运行时与编译器兼容性。
