# S2-006 — 标签拖拽与重排（实现说明）

日期: 2026-06-02

概述
- 本次任务实现了工作区标签（tab）的拖拽重排、后端持久化（`tabs.json`）与基于持久化顺序的恢复逻辑。
- 在此基础上补充了拖拽插入指示器的平滑动画，并修复了 DataTemplate 内联动画在运行时触发的 `NullReferenceException`。

实现要点
- 前端 (UI)
  - 在 `MainView.axaml` 中将标签模板的命令绑定改为直接绑定到每个 tab 的命令代理（`ActivateCommand` / `CloseCommand`），避免 DataTemplate 中对根 DataContext 的不稳定引用。
  - 使用 pointer-based 事件（已有）触发 `MainViewModel.ReorderTabs(documentId, newIndex)` 来发起重排。
  - 新增左右插入指示器：`IsDropBefore` / `IsDropAfter`，拖拽时根据指针在目标标签左/右半区决定指示位置。
  - 新增平滑动画：通过 `BoolToOpacityConverter` + `DoubleTransition` 实现插入指示器淡入淡出。
  - 修复运行时异常：将模板中的内联 `DoubleTransition` 调整为资源复用，并将过渡属性显式声明为 `Visual.OpacityProperty`，避免延迟模板构建时空引用。

- ViewModel
  - 在 `WorkspaceTabViewModel` 中新增 `ICommand ActivateCommand` 与 `ICommand CloseCommand`，增加构造重载以注入命令代理。
  - 在 `WorkspaceTabViewModel` 中新增拖拽视觉状态：`IsDropTarget`、`IsDropBefore`、`IsDropAfter`。
  - 在 `MainViewModel.SyncWorkspaceState()` 中为每个打开标签生成 `ActionCommand`（代理调用 `ActivateTab` / `CloseTab`），并通过新的构造函数注入到 `WorkspaceTabViewModel`，从而允许模板内直接绑定。

- 后端 (Workspace 服务)
  - `InMemoryWorkspaceService.MoveTab(documentId, newIndex)`：实现内存重排并在成功后将当前顺序保存到 `.muse/settings/tabs.json`。
  - `OpenWorkspace(root)`：增强恢复逻辑：
    - 优先从 `.muse/recovery/*.json` snapshots 加载（已有）。
    - 当无 recovery snapshots 但存在 persisted `tabs.json` 时，会基于 persisted ids 构造 `OpenTabs`（仅包含仍然存在的文件），从而支持 persisted-only 恢复场景。

  - `Muse.Tests/MainViewModelWorkspaceIntegrationTests.cs`（`ReorderTabs` 路径与无效文档路径覆盖）。

注意事项与边缘情况
- persisted ids 指向已删除或不可访问的文件时，恢复逻辑会忽略该 id（不抛异常）。
- recovery snapshot 与 persisted order 同时存在时，以 snapshot 为基础并按 persisted 顺序重排 snapshot 中存在的项；未出现在 persisted order 的 snapshot 会被追加在末尾。
- XAML 绑定方案已改为每-tab 命令代理，避免在 DataTemplate 中对 `ElementName` / `x:Reference` 的依赖，从而提高运行时与编译器兼容性。
- 拖拽动画在延迟模板构建阶段对资源解析更敏感；建议保持“资源定义 + 显式 AvaloniaProperty”写法，避免回退到模板内联过渡定义。

后续建议（可选）
- UI 冒烟与交互改进：手动在 `Muse.Desktop` 上验证拖拽体验（动画、拖动阈值、跨行滚动等）。
- 可扩展性：若后续需要外部扩展命令（例如插件化 close/activate），建议将代理命令替换为绑定到一个 `ITabActionService`。
- 持久化兼容性：考虑在 `tabs.json` 中保存额外元数据（如 FileName、LastTouchedAt）以减少磁盘查找并提升恢复健壮性。

# S2-007 — 图片粘贴与资产保存（实现说明）

日期: 2026-06-04

概述
- 本次任务实现“将剪贴板/拖拽图片保存到活动 Markdown 文档所在目录的 `assets`，并在编辑器中插入相对路径且在预览侧渲染”的完整流水线。解决了分屏/异步线程导致的渲染不一致问题，并加入文件去重策略与命名冲突处理。

实现要点
- 剪贴板读取：实现了基于 Win32 STA 的回退链读取 `FileGroupDescriptorW` / `FileContents` 的读取逻辑（位于 `Muse/Services/AvaloniaClipboardService.cs`），优先尝试原始文件名，遇到格式错误或控字符回退为时间戳命名。
- 资产保存：`Muse.Assets/AssetService.SaveImageAsync` 负责写盘、SHA256 去重、冲突命名策略（保留原名或生成 `pasted-<timestamp>`），并返回相对路径与绝对路径。
- 插入逻辑：`MainView.axaml.cs` 的 `HandlePasteAsync` 取得字节流、调用 `AssetService`、在光标位置插入 `![](assets/...)` 相对路径；包含重复粘贴防抖逻辑避免二次插入。
- 预览解析：`PreviewBlockViewModel.TryParseImageFromContent` 增强为多候选查找（活动文档目录、打开标签目录、repo/files、CWD、用户 Downloads 等），若找到则设置 `IsImage` 与 `ImageFilePath`。
- 绑定与线程：`PreviewBlockViewModel` 实现 `INotifyPropertyChanged`，`OnPropertyChanged` 确保在 `Avalonia.Threading.Dispatcher.UIThread` 上触发，修复分屏场景下属性变更未更新 UI 的问题。
- Bitmap 创建：为避免跨线程/文件锁导致的不可渲染，`PathToImageConverter` 在 UI 线程创建 `Bitmap`（先 ReadAllBytes 到内存流，再在 UI 线程构建 `Bitmap`），保证预览稳定渲染。
- 容错与降级：`App.Resolve<T>` 的异常在解析链中做了宽容处理（fallback implementations），避免在查询 Workspace 服务失败时中断解析流程。

- 本地验证：已执行 `dotnet build` 与运行桌面程序，粘贴/拖拽场景在分屏下通过手工验证渲染正常。

注意事项与后续建议
- Win32 剪贴板的 `FileGroupDescriptorW` 在不同来源（例如不同应用或 Office）返回格式不一致，建议在 `AvaloniaClipboardService` 中对 filename 做严格清洗（剔除控制字符）并增加回退名的可配置策略。
- 建议增加一个轻量的集成测试：在 headful 环境下 mock/注入一个可控的 `IClipboardService`，验证 `HandlePasteAsync` -> `AssetService.SaveImageAsync` -> `Preview` 渲染的端到端行为。
- 若遇到跨平台构建（非 Windows）时，应把 Win32 专用代码用运行时平台判定包裹并提供纯托管回退实现。
