# Muse 项目架构文档（Vibe Coding约束版）

## 1. 目标与原则
- 目标：在Vibe Coding开发模式下，保持高迭代速度的同时，确保代码可维护、可测试、可扩展。
- 第一原则：高内聚、低耦合。
- 第二原则：先边界后实现，先契约后编码。
- 第三原则：每次改动必须可验证、可回滚。
- 第四原则：AI只负责生成，不负责拍板；架构约束和验收门禁负责兜底。

## 2. 架构总览
### 2.1 分层架构
- 表现层（UI）：Views、交互状态、动画与主题。
- 应用层（Application）：用例编排、命令处理、事务边界。
- 领域层（Domain）：文档模型、编辑规则、核心业务策略。
- 基础设施层（Infrastructure）：文件系统、导出、RAG、外部模型调用。

### 2.2 依赖方向（强约束）
- 仅允许 UI -> Application -> Domain -> Infrastructure 抽象接口 的单向依赖。
- Domain 禁止直接依赖 UI 和具体基础设施实现。
- Infrastructure 通过接口适配 Domain/Application，不反向污染。

### 2.3 模块拆分（首版）
- Muse.Editor.Core：编辑器核心域模型与命令。
- Muse.Editor.Rendering：Markdown解析、增量渲染、视图映射。
- Muse.Workspace：文件树、多标签、自动保存、恢复。
- Muse.Assets：图片粘贴/拖拽落盘、路径策略。
- Muse.Export：HTML/PDF/Docx导出。
- Muse.ThemeUX：主题、快捷键、动效系统。
- Muse.AI.RAG：检索索引、上下文拼装、模型路由。
- Muse.Shell：应用入口、DI装配、跨模块集成。

### 2.4 当前仓库落地映射（S1-001）
- 现有 `Muse` 项目暂作为共享 UI 宿主，继续承载 Avalonia App、Views、ViewModels。
- 新增 `Muse.Shell` 项目作为组合层入口，负责后续 DI 装配与跨模块集成。
- `Muse` -> `Muse.Shell` 为当前唯一新增上行引用，平台宿主项目仍维持对 `Muse` 的引用，不做入口重构。
- 当前已创建的模块项目：`Muse.Editor.Core`、`Muse.Editor.Rendering`、`Muse.Workspace`、`Muse.Assets`、`Muse.Export`、`Muse.ThemeUX`、`Muse.AI.RAG`、`Muse.Shell`。
- 当前最小依赖关系：`Muse.Editor.Rendering` -> `Muse.Editor.Core`；`Muse.Shell` -> 各功能模块；`Muse` -> `Muse.Shell`。
- 约束：在进入 S1-002 之前，禁止把具体业务实现塞回 `Muse` 项目，`Muse` 只允许保留 UI 宿主职责。

### 2.5 当前依赖图（S1-001）
- 平台宿主（`Muse.Desktop`/`Muse.Browser`/`Muse.iOS`/`Muse.Android`）-> `Muse`
- `Muse` -> `Muse.Shell`
- `Muse.Shell` -> `Muse.Editor.Core` / `Muse.Editor.Rendering` / `Muse.Workspace` / `Muse.Assets` / `Muse.Export` / `Muse.ThemeUX` / `Muse.AI.RAG`
- `Muse.Editor.Rendering` -> `Muse.Editor.Core`

### 2.6 Core 契约基线（S1-002）
- `IDocumentModel`：定义文档身份、版本号、脏状态三项最小领域元数据。
- `ICommand`：定义编辑命令名称与针对文档的可执行判定。
- `IEditorSession`：定义会话标识、当前文档、撤销重做能力及命令执行入口。
- `CommandExecutionResult`：统一命令执行结果，显式表达成功/失败、文档版本与错误信息。
- 约束：S1-002 只定义最小接口与结果对象，不引入具体编辑实现；契约验证由独立测试工程 `Muse.Editor.Core.Tests` 承担。

### 2.7 Rendering 契约基线（S1-003）
- `RenderRequest`：统一渲染输入，当前包含 Markdown 文本、模式上下文与主题标识。
- `RenderedBlock` / `RenderedBlockKind`：定义纯 DTO 级渲染块结果，不暴露 UI 类型。
- `RenderingResult` / `RenderDiagnostic`：统一渲染成功与失败输出，便于后续增量渲染与错误观测。
- `MarkdownEngineAdapter`：作为渲染入口，内部依赖可替换的 `IMarkdownBlockParser`，避免适配器与具体解析实现强耦合。
- `IRenderingTelemetry`：为后续日志与观测预留接口，本阶段仅用于失败记录。
- 约束：Rendering 层当前只输出纯结果对象，UI 渲染映射必须在后续壳层处理，不允许在此阶段引入 Avalonia 类型。

### 2.8 三模式路由骨架（S1-004）
- UI 侧已落地 `EditorMode` 三态：`Edit` / `Split` / `Read`。
- 分屏模式已落地 `SplitOrientation` 双态：`Horizontal` / `Vertical`。
- 当前模式切换通过 ViewModel 命令触发，仅切换状态，不直接耦合渲染实现。
- 约束：
  - 模式路由只负责状态机与视图切换，不承载 Markdown 业务逻辑。
  - 分屏切换按钮仅在分屏模式可用，避免无效状态变更。
  - 后续接入渲染链路时，必须通过 Rendering 模块输出 DTO 做映射。

### 2.9 分屏状态保持基线（S1-005）
- 分屏横竖切换共用同一份 `MarkdownDraft`，确保输入内容不因布局切换丢失。
- 分屏源码输入区共用 `SplitSourceCaretIndex`，在横竖切换后保持光标上下文。
- 编辑模式、分屏模式、阅读模式当前共享同一份预览占位文本来源，确保模式切换时上下文连续。
- 约束：
  - 状态保持逻辑只落在 ViewModel 状态层，不引入平台特定控件行为依赖。
  - 后续若引入滚动位置保持，必须以可测试状态对象承载，避免直接在视图层硬编码。

### 2.10 渲染映射层基线（S2-002）
- UI 宿主层已新增 `IMarkdownPreviewService` / `MarkdownPreviewService`，统一承接 Markdown 文本到预览 ViewState 的转换。
- `MainViewModel` 当前通过映射服务消费 `RenderingResult`，不直接依赖解析器实现细节。
- 编辑模式、分屏模式、阅读模式预览已统一走同一映射出口，诊断信息以轻量提示字段暴露。
- 已新增 `Muse.Tests`，覆盖映射层模式映射、空块占位与失败诊断路径。
- 已补充 `MainViewModelWorkspaceIntegrationTests`，覆盖工作区命令触发后 UI 状态摘要与脏状态联动。
- 约束：
  - UI 仅依赖 Rendering DTO 与映射服务，不得在 ViewModel 内嵌解析逻辑。
  - 后续若扩展富文本渲染，只允许在映射层扩展，不反向污染 Rendering 契约。

### 2.11 Workspace 最小能力基线（S2-003）
- `Muse.Workspace` 已落地最小状态模型：`WorkspaceState`、`WorkspaceTabState`、`FileTreeNode`。
- 已落地最小契约：`IWorkspaceService`、`IAutoSaveScheduler`。
- 已提供内存实现：`InMemoryWorkspaceService`、`MemoryAutoSaveScheduler`，支持工作区切换、标签激活、脏状态标记与自动保存调度触发。
- 已新增独立测试工程 `Muse.Workspace.Tests`，覆盖最小状态流转与调度行为。
- UI 宿主已接入最小联调入口：加载当前工作区与打开 Sprint2 任务卡，并展示工作区状态摘要。
- 分屏区域已展示活动文档与脏状态标记，便于后续自动保存与编辑态联动扩展。
- 已补充最小保存链路：`IWorkspaceService.SaveDocument` 可将活动文档从脏状态回落到已保存状态。
- UI 宿主已接入“保存当前文档”命令，完成最小闭环验证。
- 已补充保存结果契约：`SaveDocumentResult` 统一表达保存成功/失败、错误码与错误信息。
- UI 宿主已接入保存反馈提示（成功/失败），避免静默失败。
- `SaveDocument` 现已接收待保存内容并执行真实文件写入，失败统一映射为 `io_error` 等错误码。
- 打开默认任务文档时，UI 会主动读取活动文档内容到编辑区，并通过状态保护避免加载过程误触发脏标记。

## 3. AI开发防跑偏约束（必须执行）
## 3.1 单次任务粒度
- 一个任务只允许改动一个主模块，跨模块改动必须拆分子任务。
- 每个任务必须声明：输入、输出、影响范围、回归范围。

## 3.2 代码生成契约
- 必须先有接口定义，再生成实现。
- 必须先有最小可运行测试，再补全功能代码。
- 禁止在未声明依赖的情况下引入新三方库。
- 禁止跨层直接调用（例如UI直接读写文件系统）。

## 3.3 目录与命名约束
- 新增代码必须放入所属模块，不允许“临时工具类”散落。
- 命名要求语义清晰，禁止 GenericHelper、Util2、TempService 等模糊命名。
- 所有配置项集中到配置模型，禁止魔法字符串硬编码分散。

## 3.4 复杂度上限
- 方法行数目标小于60行；超过需拆分。
- 类职责不超过3个核心行为；超过需重构。
- 单文件变更超过300行时必须拆任务并说明原因。

## 4. 关键技术决策（ADR）
- ADR-001：Markdown兼容策略以“Typora行为一致性优先”。
- ADR-002：模式策略固定为编辑模式（默认）/分屏模式（左右或上下）/阅读模式。
- ADR-003：导出高级选项由导出菜单驱动，不在正文中嵌入元数据。
- ADR-004：AI模型采用本地与云双路，由用户设置菜单选择。
- ADR-005：主题系统兼容Typora CSS导入，默认液态玻璃风格并强制可读性校验。

## 5. 质量门禁（Gate）
## 5.1 提交前门禁
- 编译通过。
- 新增或变更功能必须有对应测试。
- 关键路径（编辑、保存、导出）冒烟通过。
- 变更说明包含：风险点、回滚方案、影响模块。

## 5.2 合并前门禁
- 通过静态分析与格式检查。
- 通过模块边界检查（禁止反向依赖）。
- 通过回归测试集（至少核心编辑+导出+资源管理）。

### 5.2.1 当前边界检查脚本
- 当前仓库已提供脚本：[scripts/Test-ArchitectureBoundaries.ps1](scripts/Test-ArchitectureBoundaries.ps1)
- 检查对象：所有 `.csproj` 的 `ProjectReference`。
- 检查规则：仅允许当前架构文档中声明的白名单依赖方向；发现未授权引用即返回失败。
- 当前覆盖：UI 宿主层、平台宿主层、Shell 组合层、Core/Rendering 功能层、测试工程。

### 5.2.2 当前本地质量门禁脚本
- 当前仓库已提供脚本：[scripts/Invoke-QualityGate.ps1](scripts/Invoke-QualityGate.ps1)
- 当前执行顺序：
  - 解决方案构建
  - 架构边界检查
  - Core 契约测试
  - Rendering 契约测试
  - Workspace 契约测试
  - UI 映射测试
  - 桌面冒烟构建
- 约束：后续新增模块测试或关键冒烟步骤时，必须同步更新该脚本，避免门禁与真实质量要求脱节。

## 5.3 发布前门禁
- 性能验收：启动小于1秒（标准开发机基线）。
- 体验验收：动画平滑，输入无阻塞感。
- 可用性验收：大文档、图片密集、频繁切模式场景稳定。

## 6. Vibe Coding工作流（标准流程）
1. 定义任务卡：目标、边界、验收标准、回滚点。
2. 生成接口草案：先约定输入输出与异常语义。
3. 编写最小测试：先红后绿。
4. AI生成实现：仅限当前任务范围。
5. 本地验证：编译、测试、冒烟、性能快测。
6. 记录文档：更新本文件与对应功能文档。
7. 人工复核：重点检查跨层依赖、隐式状态、副作用。

## 7. 防AI乱写清单（评审Checklist）
- 是否出现跨层依赖或循环依赖。
- 是否引入与需求无关的重构或大规模格式化。
- 是否增加未评审的新库或新协议。
- 是否出现隐式全局状态。
- 是否缺少失败路径处理与错误提示。
- 是否未更新文档、配置说明、迁移说明。

## 8. UI与体验约束（apple-design落地）
- 主题：浅色/深色默认液态玻璃风格。
- 可读性：组件与文字对比度必须达标，禁止低对比配色上线。
- 动效：光标平滑移动，弹出/关闭淡入淡出，页面切换平滑。
- 反馈：加载状态优先使用三点闪烁，降低等待焦虑。
- 布局：卡片化、内容优先，避免视觉噪声。

- 所有开发文档放在 files 目录。
- 所有方案文档单一文件持续维护，不再创建版本副本。
- 每个模块完成后必须同步更新对应文档与实施计划状态。

  - 临时方案影响范围与失效时间。
  - 后续偿还技术债时间点。
